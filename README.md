# Hilads

> Open the app. See who's around. Jump into something happening now.

Hilads is a real-time social app. Join a city, see who's online, chat, discover events, and create your own — no sign-up required.

---

## What it does

- **City chat** — public real-time chat, one channel per city
- **Hot screen** — active events in your city (one-shot + recurring venues)
- **Events** — create, join, edit, and delete events; every event has its own chat
- **Presence** — who's online right now, live join/leave updates
- **People here** — browse online users, view profiles
- **My Events** — events you created, with recurring series deduplicated
- **Direct messages** — 1:1 private chat for registered users
- **Notifications** — in-app bell + web push for DMs, event messages, and new events

---

## How it works

```
Open app
  → geolocation resolves your city (or pick manually)
  → see city chat, online users, active events
  → jump in — no account needed
  → create an event or DM if you want more
```

**Guest session:** instant access, UUID + nickname, no sign-up.
**Registered account:** persistent profile, DMs, ownership across devices.

## Events

Two types of events:

### One-shot events
- Created by any user (guest or registered)
- Custom title, location hint, start → end time
- Expire automatically
- Own real-time chat, deleted when the event ends

### Recurring events
- Seeded per city: bars, coffee shops, curated venues
- Daily/weekly schedule
- Show "↻ Recurring" badge
- Only today's occurrence is shown in Hot

### Event ownership
Every event is owned by its creator (tracked via guest UUID or registered user ID).

Creators can:
- Edit title, location, time
- Delete the event (soft delete — disappears from all views immediately)

Creator UX:
- "👑 Your event" badge in the event header
- "✏️ Edit event" CTA replaces the Join button
- Owner-prompt card in event chat on first open

## Architecture

```
apps/
  web/          React frontend (mobile-first)
backend/
  api/          PHP REST API (nginx + FPM via Docker)
  ws/           Node.js WebSocket server
docs/
  mvp.md        Full product definition
  design.md     UX and visual direction
infra/          Infrastructure notes
render.yaml     Render service definitions
```

**Backend** is intentionally thin: router → repositories → PDO → PostgreSQL. No framework, no ORM.

**WebSocket server** handles two things: presence snapshots (join/leave) and real-time message push. PHP calls it via internal HTTP on a fire-and-forget basis.

**Events** come from two sources:
- User-created (one-shot, any user — guests or registered)
- Recurring series (curated venues, seeded per city, daily occurrences generated by cron)

**Event ownership** is tracked on `channel_events` via two columns:
- `guest_id` — always set (guest UUID from cookie)
- `created_by` — set for registered users (persistent user UUID)

**Schema migrations** are applied idempotently inside `Database::pdo()` on first connection — no migration runner required.

**Message retention** is handled by a daily cleanup job (`POST /internal/cleanup`):
- City chat: today only
- Event chat: deleted when event expires (+1h)
- DMs: 7-day history

## Running Locally

### Backend API

```bash
cd backend/api
cp .env.example .env
# fill: DATABASE_URL, R2_*, WS_INTERNAL_URL, MIGRATION_KEY

docker build -t hilads-api .
docker run -p 8080:80 --env-file .env hilads-api
```

### WebSocket server

```bash
cd backend/ws
npm install
node server.js
# WS on :8081 (public), HTTP on :8082 (internal PHP broadcasts)
```

### Frontend

```bash
cd apps/web
npm install
cp .env.example .env
# set VITE_API_URL, VITE_WS_URL

npm run dev
# http://localhost:5173
```

## Scheduled Jobs (Cron)

Two daily jobs, both protected by `MIGRATION_KEY`:

**Generate recurring event occurrences** (2:00 AM UTC)
```
POST /internal/event-series/generate?key=YOUR_KEY
```

**Cleanup stale messages + expired channels** (3:00 AM UTC)
```
POST /internal/cleanup?key=YOUR_KEY
```

Set these up as Render Cron Jobs or via [cron-job.org](https://cron-job.org).

## Seeding Recurring Venues

Hilads bootstraps cities with recurring events (bars, coffee shops) so the Hot screen is never empty on launch.

The venue list lives in `backend/api/src/venues_seed.php`. Each venue becomes a daily recurring `event_series` row. The endpoint is idempotent: re-running it skips already-created series (deduplicated via `source_key`).

### Run a dry run first

```bash
curl -X POST "https://your-api.onrender.com/internal/seed-static-venues" \
  -H "X-Api-Key: YOUR_MIGRATION_KEY" \
  -H "Content-Type: application/json" \
  -d '{"dryRun": true}'
```

### Run the real import

```bash
curl -X POST "https://your-api.onrender.com/internal/seed-static-venues" \
  -H "X-Api-Key: YOUR_MIGRATION_KEY" \
  -H "Content-Type: application/json" \
  -d '{"dryRun": false}'
```

After seeding, generate today's occurrences:

```
POST /internal/event-series/generate?key=YOUR_MIGRATION_KEY
```
