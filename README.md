# Hilads

> Open the app. See who's around. Jump into something happening now.

Hilads is a real-time social app. You join a city, see who's online, chat, and discover events and places — no sign-up required.

---

## What it does

- **City chat** — public real-time chat, one channel per city
- **Hot screen** — active events in your city (one-shot + recurring venues)
- **Presence** — who's online right now, live join/leave updates
- **People here** — browse online users, view profiles
- **Direct messages** — 1:1 private chat for registered users
- **Event chats** — every event has its own dedicated chat channel
- **Recurring venues** — bars and cafés seeded per city with daily schedules
- **Notifications** — in-app bell + web push for DMs, event messages, and new events (registered users)

---

## How it works

```
Open app
  → geolocation resolves your city (or pick manually)
  → see city chat, online users, active events
  → jump in — no account needed
  → create a DM or event if you're registered
```

Guest session: instant access, UUID + nickname, no sign-up.
Registered account: persistent profile, DMs, recurring event creation.

## Tech Stack

| Layer | Stack |
|---|---|
| Frontend | React 18 + Vite (mobile-first) |
| Backend | PHP 8.2 — plain REST API, no framework |
| Database | PostgreSQL |
| Real-time | Node.js WebSocket server + 3s poll fallback |
| Media storage | Cloudflare R2 |
| Hosting | Render (API + WS) · Vercel (frontend) |

## Architecture

```
apps/
  web/          React frontend (mobile-first)
backend/
  api/          PHP REST API (nginx + FPM via Docker)
  ws/           Node.js WebSocket server
docs/
  mvp.md        Full product definition
  design.md     UX and visual direction
infra/          Infrastructure notes
render.yaml     Render service definitions
```

**Backend** is intentionally thin: router → repositories → PDO → PostgreSQL. No framework, no ORM.

**WebSocket server** does two things: presence snapshots (join/leave) and real-time message push. PHP calls it via internal HTTP.

**Events** come from two sources:
- User-created (one-shot, any user)
- Recurring series (curated venues, seeded per city, daily occurrences generated by cron)

**Message retention** is handled by a daily cleanup job (`POST /internal/cleanup`):
- City chat: today only
- Event chat: deleted when event expires
- DMs: 7-day history

## Running Locally

### Backend API

```bash
cd backend/api
cp .env.example .env
# fill: DATABASE_URL, R2_*, WS_INTERNAL_URL, MIGRATION_KEY

docker build -t hilads-api .
docker run -p 8080:80 --env-file .env hilads-api
```

### WebSocket server

```bash
cd backend/ws
npm install
node server.js
# WS on :8081 (public), HTTP on :8082 (internal PHP broadcasts)
```

### Frontend

```bash
cd apps/web
npm install
cp .env.example .env
# set VITE_API_URL, VITE_WS_URL

npm run dev
# http://localhost:5173
```

## Scheduled Jobs (Cron)

Two daily jobs, both protected by `MIGRATION_KEY`:

**Generate recurring event occurrences** (run at 2:00 AM UTC)
```
POST /internal/event-series/generate?key=YOUR_KEY
```

**Cleanup stale messages + expired channels** (run at 3:00 AM UTC)
```
POST /internal/cleanup?key=YOUR_KEY
```

Set these up as Render Cron Jobs or via [cron-job.org](https://cron-job.org).

## Principles

- Mobile-first, always — no web patterns
- No sign-up friction — guest mode is first-class
- Simplicity over engineering — no framework, no ORM, no abstraction layers
- One product rule: *does this make the city feel more alive right now?*
